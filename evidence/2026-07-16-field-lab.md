# Field-lab evidence — 2026-07-16

This is a curated, secret-free summary. Raw JSON remains ignored under `.runs/`. Synthetic
facts and ephemeral resource identifiers are intentionally omitted here.

## Environment

- Hosted API: `https://api.supermemory.ai`, one promotional/hosted account.
- Client location/timezone: India; wall time includes WAN, TLS, and client overhead.
- LLM path: OpenRouter with the configured small OpenAI model for agent/Router checks.
- Local server: fresh Supermemory server v0.0.5, disposable data directory, local
  `Xenova/bge-base-en-v1.5` embeddings, remote OpenAI-compatible extraction model.
- SMFS Bash tool: Python package v0.0.2, disposable hosted container.
- Sample sizes are deliberately small; these are contract/behavior probes, not load tests.

## Hosted core API — two runs

| Probe | Observation |
|---|---|
| Direct memory create | Three exact memories created; second run client wall 1,068 ms. |
| Memory search | Correct launch fact ranked first; server timing 311 ms on the initial second-run call. |
| Hybrid search | Retrieved exact memories; after document ingest it returned a raw chunk plus memories. |
| Default v4 mode | Omitting `searchMode` returned memories only, including after a matching document existed. |
| Profile | Stable facts in `static`, dated event in `dynamic`, query results attached; `preferences` bucket empty. |
| Isolation | Same query in a fresh container returned zero results; server timing 77 ms. |
| Version update | New fact was version 2 with the original as parent/root; current search returned the new value. |
| Single forget | Returned `forgotten=true`; forgotten fact was absent from the next search. |
| Mass-forget dry-run | First run ~5.8 s and found one candidate; second run exceeded 60 s. First response wording said “Forgot 1” despite dry-run, so inspect structured fields. |
| Document ingest | Queued immediately; processing to `done` took ~21.1 s and ~18.2 s. |
| v3 document search | Exact synthetic source chunk returned; score ~0.722; server timing 142–184 ms. |
| `customId` upsert | Same document ID returned. Reprocessing took ~32.5 s and ~24.5 s. One run added only the new fact; the other returned all four re-extracted facts, so do not assume stable incremental extraction. |
| Conversation upsert | Initial and appended lists returned the same document ID, with `queued` then `extracting`. |
| Cleanup | Document delete succeeded; following GET returned expected 404. |
| Search latency | Three-sample client median was ~591 ms in run 1 and ~1,084 ms in run 2; reported server work was generally much lower. |
| Personalized agent | Profile-aware two-line response used the old date and concise preference; complete call ~4.1 s including retrieval and LLM. Best-effort conversation persistence succeeded. |

The first memory-update attempt omitted `containerTag` by following a prose example and was
rejected. Adding the container succeeded. The generated TypeScript SDK requires the field.

## Memory Router — two runs plus late control

| Probe | Observation |
|---|---|
| Forwarding | OpenRouter completions succeeded through the proxy in ~1.2–2.4 s client wall. |
| Diagnostic signal | Initial calls returned context-modified and conversation-ID headers. Some fresh-conversation calls returned no diagnostic headers. |
| Initial write | Model acknowledged the synthetic phrase. Direct v4 hybrid search later found the raw user-message chunk. |
| Same conversation ID | A new question returned “Acknowledged.” again instead of the phrase, reproduced twice. |
| Same user, new conversation | Could not recall the phrase after 20 seconds; a later retry also failed. |
| Memory visibility | Hybrid search found one raw chunk; memories-only search found zero distilled memories. |
| Full-history control | Supplying the original fact in the explicit message list answered correctly. |
| Other-user control | Returned “Unknown.” and did not leak the phrase. |

Interpretation: forwarding and isolation worked, but transparent continuation/cross-session
memory did not work for this workload. This does not establish the root cause. It is a
release gate for any Router adoption.

## Connector

Creating a one-page web-crawler connection against `example.com` returned `403` with a
message that Scale plan or higher is required. No connector resource was created. This
verifies entitlement behavior, not crawler quality.

## Self-hosted server

- Installer release checksum was verified.
- Worker became ready in approximately 15 seconds on first boot after local model setup.
- The server logged approximately 1.7 GB baseline memory and 1 GB ingestion headroom.
- A text document reached `done` and produced two intended memory facts.
- Hybrid search returned the two memories and source chunk; server timing was 31 ms on the
  tiny local dataset.
- Profile returned both intended stable facts.
- Total scripted boot/ingest/search/profile path was approximately 24.5 seconds.

This was a fresh install, not an upgrade, durability, backup/restore, concurrency, or large
document test.

## SMFS Python Bash tool

- Writing a synthetic `memory.md`, reading it, listing it, semantic `sgrep`, and cleanup all succeeded.
- Semantic search found the file in approximately 10.3 seconds.
- Immediate `profile.md` still reported no extracted memories, consistent with eventual processing.
- `ls -la` showed zero size for a file that `cat` could read, an alpha implementation quirk.
- This is a virtual Bash parser, not unrestricted operating-system command execution.
